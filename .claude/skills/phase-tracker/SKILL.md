# Agent OS Phase Tracker

Load this skill before doing implementation work in this repo. Its job is to stop you from either re-implementing something that already exists, or building ahead into a phase that hasn't started yet.

**This file is a snapshot, not a constant.** Treat the status below as true as of when it was last updated (see the date at the bottom), but if you complete a deliverable, finish a phase, or otherwise change the status while working in this repo, come back and update this file in the same session ‚Äî that's what keeps it trustworthy for next time. A stale phase tracker is worse than no phase tracker, because it gives false confidence.

## How to use this when working

Before implementing a feature, fixing a stub, or scoping out work: check which phase it belongs to using the tables above. If it's a Phase 4 or 5 item, don't build it during Phase 3 work ‚Äî flag that it's out of sequence instead of quietly doing it anyway, since skipping ahead tends to mean building against interfaces/decisions that haven't been finalized yet (e.g. the approval engine and conflict detection in Phase 3 Day 5 inform what Phase 4 hardening actually needs to test).

If a deliverable listed as "stubbed" or "not implemented" above turns out to already have real code when you check, the tracker is out of date ‚Äî update it rather than trusting it blindly.
