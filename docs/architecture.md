# Architecture

## System design

Agent OS is built around a single idea: there should be exactly one place that knows the true state of your multi-agent work, and everything else — the CLI you type into, the dashboard you watch, the agent processes doing the actual coding — should be a client of that one place. That place is the gateway, a FastAPI application that binds to `127.0.0.1:47821` and never anywhere else. It owns the only SQLite connection that ever writes. It is the only thing that creates a Git worktree, the only thing that decides whether a risky shell command gets auto-approved or routed to a human, and the only thing that knows when an agent's credential has expired.

Everything upstream of the gateway — the CLI (`apps/cli`, TypeScript, Commander-based) and the dashboard (`apps/dashboard`, Next.js) — talks to it exclusively over HTTP for request/response operations and Server-Sent Events for anything that needs to update live. Neither ever opens the SQLite file directly, and neither ever spawns an agent process directly. This isn't an accident of how the code happened to get written; it's the load-bearing constraint that makes the rest of the design possible. If the CLI could write to the database directly, you'd need to reconcile two writers. If the dashboard could spawn agent processes, you'd need process-lifecycle logic in two places instead of one. Keeping the gateway as the sole source of truth means every other component can be thin, replaceable, and — critically for a personal tool you might rewrite the dashboard for twice — disposable.

Below the gateway sits the adapter layer (`packages/gateway/src/agentos_gateway/adapters/`), which is where the actual agent CLIs (`claude`, `codex`, or anything else you point Agent OS at) get wrapped behind a single interface: `start_session` → `send_task` → `stream_events` → `submit_result` → `stop`. The gateway's orchestration code — mission planning, task routing, conflict detection — never branches on which agent it's talking to. It calls the interface. Adapters are responsible for the messy, agent-specific parts: how to invoke the CLI, how to parse its output into a normalized event stream, and how to recognize when that output means "your subscription expired" versus "you're being rate-limited" versus "everything's fine, keep going."

## The five core design decisions

**FastAPI over a Node.js gateway.** The gateway needs to do three things well: serve a stable HTTP API, stream events over SSE without fighting the runtime, and manage subprocess lifecycles for spawned agent CLIs. Python's `asyncio` ecosystem treats all three as first-class — SSE is a few lines with `StreamingResponse`, and `asyncio.create_subprocess_exec` is built for exactly the job of supervising a long-running child process and reading its stdout/stderr concurrently. Node could do this too, but Python's subprocess and async-iterator primitives map onto "spawn an agent CLI and stream its output" more directly, and it keeps the orchestration layer in the same language as the agent CLIs it's likely to be inspecting and parsing.

**SQLite over a server-backed database.** Agent OS is explicitly a single-user, local-first tool — there's one developer, one machine, one gateway process. A client-server database would add an operational dependency (a database server to install, configure, and keep running) for a workload that's fundamentally single-writer and runs entirely on localhost. SQLite gives you ACID guarantees, a single file you can back up by copying it, and zero configuration, in exchange for a constraint (one writer) that the architecture already enforces by design anyway.

**Git worktrees over Docker for workspace isolation.** Each task an agent works on needs its own isolated working copy, branched from a common base, so two agents can work in parallel without stepping on each other's uncommitted changes. Docker containers would give you isolation too, but at the cost of a daemon dependency, image management, and volume-mounting complexity for what is, underneath, just "give this agent its own branch and directory." `git worktree` does exactly that — a real working directory, a real branch, sharing the same object store as the main repo — with none of the container runtime overhead, and it's available wherever Git is, which is everywhere this tool is meant to run.

**The OS keychain for credentials, never SQLite, logs, or context packs.** Credentials are the one piece of state in this system that must never end up in anything that gets persisted broadly, displayed in a UI, or handed to another agent during a task handoff. Routing every credential read and write through `keyring` (the OS-native credential store — Keychain on macOS, Credential Manager on Windows, libsecret on Linux) means there is exactly one place credentials can live, and that place is specifically designed not to leak into backups, version control, or process memory dumps the way a SQLite row or a log line would.

**SSE over WebSockets for live updates.** The dashboard and CLI need to know about new events as they happen — a task starting, an approval request appearing, a mission completing — but they never need to send anything back over that same channel; all writes go through normal HTTP requests. That's a one-directional, read-mostly access pattern, which is exactly what Server-Sent Events is for. WebSockets would add the complexity of a bidirectional protocol (framing, ping/pong keepalive, reconnection semantics on both ends) to solve a problem that's actually one-directional. SSE's automatic reconnection and plain-HTTP transport make it the simpler tool for a simpler problem.

## Data flow: from objective to merged branch

A mission's life starts with a single sentence — the objective — handed to `POST /missions`. At this point nothing has been decomposed yet; the mission exists with `status: planning` and no tasks. Calling `POST /missions/{id}/plan` is what actually does the work: `MissionService` gathers the currently registered agents (their roles and declared capabilities), builds a prompt describing the objective and those agents, and sends it to `claude-sonnet-4-6` asking for a task breakdown as a JSON DAG — task titles, which agent each should go to, what depends on what, an estimated risk level, and which files each task is expected to touch. The response gets validated (no cycles in the dependency graph, every `assigned_agent_id` actually exists) before any task rows get written, and the mission's aggregate `risk_level` is set to the highest risk level among its tasks.

Calling `POST /missions/{id}/run` is where tasks actually start moving. For every task whose dependencies are already satisfied and which has an assigned agent, the gateway creates a Git worktree on a fresh branch (`ag/{task_id}/{agent_id}`) forked from the repo's current base branch, opens an `AgentRun` row, and (once a process spawner exists — see the adapters doc for the current state of that) would hand the task to the adapter for that agent. Right now, this step does the bookkeeping a spawner needs — worktree, run row, status transition, `task.started` event — without yet invoking a real subprocess; wiring that invocation up is the largest piece of work left after this release.

While an agent is (conceptually, or once spawning exists, actually) working, it writes messages into `agent_messages`, which is what makes task handoff possible later: if a credential fails or a human reroutes the task, `RunManager.build_handoff_pack` assembles every prior message plus a live `git diff` of the outgoing branch into a single context-pack string, sanitizes it for anything secret-shaped, and hands it to whichever agent picks the task up next — so the new agent inherits the actual partial work and conversation history, not just a one-line task description.

When a task's work looks done, `GET /tasks/{id}/diff` shows what changed, and `POST /tasks/{id}/merge` folds the worktree's branch back into the base branch — but only if the main repo is currently checked out on that base branch; the gateway refuses to merge rather than silently switching your active branch out from under you. If the work isn't good, `POST /tasks/{id}/rollback` discards it and restores the last clean state instead.

## Event system: adapter to SQLite to SSE to dashboard

Every state change worth knowing about — a mission starting, a task being assigned, a credential expiring, a conflict being detected — flows through exactly one function: `event_ledger.emit()`. Nothing else writes to the `events` table, and nothing else is the source of an SSE message. `emit()` fans a single event out to three places at once: a row in SQLite (so `GET /events` can serve paginated history with filters), a line appended to a JSONL file under `.agentos/runs/` (a flat, durable log independent of the database), and a push to `SSEManager`, which immediately forwards it to every currently-subscribed SSE connection whose filters (`mission_id`, `run_id`, `agent_id`) match.

This is also why events are append-only — there's no `UPDATE` or `DELETE` path for the `events` table anywhere in the codebase. If something needs correcting (a conflict gets resolved, a credential gets rotated), the correction is itself a new event (`conflict.resolved`, `credential.rotated`), and anything reading the event stream — including the inbox's conflict list, which determines "still open" by checking whether a later `conflict.resolved` event references a given `conflict.detected` event's id — reconstructs current state by replaying the log rather than mutating a row in place.

On the consuming side, the dashboard subscribes to `GET /events/stream` with `EventSource` and gets pushed new events the moment `emit()` fires; the CLI's `agentos logs --follow` does the same thing from the terminal. Anything that wants a snapshot rather than a live feed — the inbox, the dashboard's initial page load — calls `GET /events` instead, which reads the SQLite-backed history. The two paths are deliberately the same underlying data, just accessed differently depending on whether the caller wants "what's happening now" or "what already happened."

## Security model

Three things in this system are specifically designed to never leak: credential values, in-flight secrets accidentally captured in agent output, and unreviewed risky actions.

Credential values live exclusively in the OS keychain via `keyring`. `CredentialManager.store()`/`retrieve()` are the only functions in the entire codebase that ever see a raw credential value, and they pass it straight through to/from the keychain without it touching a SQL statement, a log call, or an event payload anywhere in between. The event system enforces this independently as a second line of defense: `_sanitize_payload()` rejects any event payload containing a key that looks like `password`, `secret`, `token`, `api_key`, `credential`, or similar, so even a programming mistake elsewhere can't get a credential into the append-only event log.

Secrets that show up in agent *output* — a `claude` or `codex` process accidentally echoing an API key it found in a file it read — are a different threat, since they're free text rather than a labeled field. `RunManager._sanitize_text()` exists specifically for this: before any prior agent message gets interpolated into a handoff context pack for a fallback agent, it's scanned against four secret-shaped patterns (Anthropic keys, OpenAI keys, GitHub personal-access tokens, and long base64 blobs) plus a key:value redaction pass for anything that looks like a blocked field name written inline, and any match is replaced with `[REDACTED]` before the pack is ever persisted or handed to another agent.

Risky actions — anything an agent wants to do that touches a sensitive path, runs a recognized-dangerous command, or operates in a `production` environment — go through `ApprovalEngine.intercept()` before they're allowed to happen. Scoring is deterministic: a base weight per action type, plus penalties for touching paths matched against `policies.yml`'s `sensitive_paths` glob list, plus a flat penalty for low-reversibility action types (`deploy`, `migration`, `db`), capped at 100 — except a literal match against the `dangerous_commands` blocklist, which always scores 100 regardless of anything else. A score under 30 auto-approves; 30–99 creates a pending `action_requests` row that surfaces in `agentos inbox` / the dashboard's inbox page and blocks the calling agent for up to five minutes waiting on a human decision; 100 blocks immediately and never reaches a human at all. `policies.yml` is re-read from disk on every single call rather than cached, specifically so that editing your sensitivity rules takes effect on the very next action without requiring a gateway restart.
